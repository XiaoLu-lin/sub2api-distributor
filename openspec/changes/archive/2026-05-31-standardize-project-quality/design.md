## Context

`sub2api-distributor` 是一个独立的分销商结算门户，当前已经实现了分销商端、运营端、主库复用、邀请返利展示和线下打款流转等 MVP 能力。最近几轮迭代已经增加了邀请元信息接口、运营侧分销商启停、后台式前端路由和本地 CORS 联调支持，但项目级保障还不完整：

- 根文档和测试文档对前端端口、联调地址和演示账号的描述不一致。
- `frontend/README.md` 仍是默认 Vite 模板，不能指导项目维护者。
- 前端没有自动化测试，后端只有少量逻辑测试。
- 后端核心模块的导出类型和函数大多没有说明性注释。
- API 验收脚本没有覆盖新增的 `invite-meta` 和运营侧用户搜索能力。

这次变更不修改核心业务流程，也不改后端接口协议；目标是把已有实现整理成一套更稳定的交付基线。

## Goals / Non-Goals

**Goals:**

- 统一项目文档中的启动方式、默认端口、测试命令和演示数据说明。
- 为前端建立轻量测试基线，并让验证命令可以稳定运行。
- 扩充后端测试，覆盖认证、配置和关键纯逻辑。
- 为后端导出 API 和关键业务逻辑补齐清晰注释。
- 更新验收脚本，让它覆盖当前已存在的关键接口面。
- 通过可重复执行的验证命令证明上述改动已经生效。

**Non-Goals:**

- 不引入新的 UI 组件库、状态管理库或 CI 平台能力。
- 不重构现有业务架构，不新增数据库表或改动提现业务状态机。
- 不改动当前前后端接口 URL、字段契约和登录流程。
- 不追求高覆盖率指标，只建立足够实用、可延续的测试基线。

## Decisions

### 1. 用“一套统一文档入口”收敛项目说明

保留并重写现有 `README.md`、`TESTING.md` 和 `docs/distributor-system-overview.md`，同时替换掉默认的 `frontend/README.md`。这样能最小化路径变动，避免新增一套重复文档入口。

备选方案：

- 新增单独的 `docs/project-handbook.md`
  - 缺点是信息容易再次分散。
- 只修 README
  - 缺点是测试说明和系统说明仍然漂移。

### 2. 前端测试采用 `vitest` 轻量落地

前端当前是 Vite + TypeScript 项目，`vitest` 与现有构建栈兼容最好，接入成本最低。测试范围聚焦在无需复杂 DOM 环境的基础模块：

- `src/utils/format.ts`
- `src/session/storage.ts`
- `src/api/http.ts`
- `src/config/navigation.ts`

这样可以先建立稳定、可执行的测试入口，而不把范围扩到复杂页面渲染和 E2E。

备选方案：

- 不新增依赖，只做文档和 build 校验
  - 无法满足“补测试基线”的目标。
- 直接接入 Playwright/Cypress
  - 过重，超出本轮范围。

### 3. 后端测试优先补“纯逻辑 + 安全边界”

后端以 Go 标准测试体系为主，优先补这些低耦合、高价值点：

- `auth.ParseToken` 的签发与校验行为
- `config.Load` 的默认值和环境变量覆盖逻辑
- 已有 `distributor` 纯逻辑函数
- 已有 `app` CORS 白名单逻辑

不在这一轮引入数据库 mock 框架，避免为了覆盖率引入额外维护成本。

备选方案：

- 为所有 service 方法补数据库级测试
  - 需要额外 mock/测试容器，超出本轮节奏。

### 4. 后端注释采用“导出符号必注释 + 复杂逻辑补解释”的策略

注释目标不是机械补满每一行，而是让维护者快速理解：

- 每个导出类型、常量、函数的职责
- 关键事务逻辑、金额计算逻辑、权限路由的意图
- 错误变量对应的语义边界

这能在不制造注释噪音的前提下显著提升可读性。

### 5. 验收脚本扩展为“当前接口面回归”

现有 `scripts/api_acceptance.sh` 已经能串起主要流程，本轮直接在其基础上补：

- `GET /portal/invite-meta`
- `GET /ops/users/lookup`

这样可以保持一份真实联调用的脚本，而不是维护多份分散脚本。

## Risks / Trade-offs

- [新增 `vitest` 依赖] → 只引入一项轻量测试依赖，不追加 UI 测试生态，控制复杂度。
- [文档与演示账号再次漂移] → 文档明确以 `seed_demo_data.sh` 生成的数据和当前推荐端口为准，并把历史端口说明降为兼容信息。
- [后端注释过多变成噪音] → 只给导出符号和关键逻辑加注释，不对显而易见的赋值和分支逐行注释。
- [测试范围仍不覆盖数据库查询细节] → 在文档中明确这是 v1 质量基线，后续如要继续补 service/handler 层集成测试再单独开 change。

## Migration Plan

1. 先补 OpenSpec 工件，固定变更边界和任务。
2. 引入前端测试依赖并添加最小测试用例。
3. 扩充后端测试文件并补充注释。
4. 统一文档与验收脚本。
5. 运行后端测试、前端测试、前端构建和 API 验收命令。

这次变更不涉及数据库结构迁移，也不涉及线上兼容性切换。
